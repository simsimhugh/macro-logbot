# 개발 프로세스: AI DLC v2

> **이 문서의 정체성**: AI DLC 의 전체 흐름과 분업 구조를 정의하는 실행 매뉴얼. 코드 레벨 상세는 각 파일 링크 참조.

## 1. 핵심 원칙 — Mechanical / Semantic 분업

| layer | 담당 | 토큰 | 상세 |
|---|---|---|---|
| **Mechanical** | GitHub (Actions + Security) | 소비 없음 | lint, type, test, SAST, secrets, deps, Dependabot, Push Protection |
| **Semantic** | 4 AI reviewer | 사용 | 설계, 로직 결함, 비즈니스 보안, edge case — **GitHub 이 이미 하는 영역 검토 금지** |

→ lint/CVE/secret 등 mechanical 검증은 GitHub 에 위임하여 AI 토큰 소비를 제거한다.

> **용어 정리**
>
> | 용어 | 의미 |
> |---|---|
> | **lint** | 코드 스타일 · 문법 오류 자동 검사 (ruff, eslint) |
> | **type** | 정적 타입 검사 (mypy, tsc) |
> | **test** | 자동화 테스트 실행 (pytest) |
> | **SAST** | Static Application Security Testing — 소스 코드 정적 보안 분석 (bandit, semgrep, CodeQL) |
> | **secrets** | 소스 코드 내 API key · password · token 유출 탐지 (gitleaks) |
> | **deps** | dependency 의 알려진 취약점(CVE) 검사 (pip-audit) |
> | **CVE** | Common Vulnerabilities and Exposures — 공개된 보안 취약점 식별 번호 |
> | **Dependabot** | GitHub 내장 — dependency CVE 자동 감지 + fix PR 생성 + 정기 버전 업데이트 |
> | **Push Protection** | GitHub 내장 — push 시점에 secret 유출을 차단 (commit 자체 거부) |

## 4. Semantic layer — 4 AI Reviewer

### 4.1 Reviewer scope

| agent | scope | GitHub 영역 제외 |
|---|---|---|
| **architect** | 설계 · 의존성 · 추상화 | — |
| **code-reviewer** | 로직 결함 · SOLID · design smell | syntax/style/lint/type/dead code |
| **security-reviewer** | 비즈니스 보안 · TOCTOU · auth/authz | CVE/secret/dependency/SAST |
| **test-engineer** | edge case · 새 test 권고 · 전략 | test 실행/coverage %/mechanical |

### 4.2 Verdict 정책

verdict 는 finding severity 기반 role-specific 자동 산출 (agent 자율 결정 금지). 정책 표·계산 규칙은 [`post-review/SKILL.md`](../../.claude/skills/post-review/SKILL.md) § "verdict 자동 결정" (+ `post.sh` `expected_verdict()`) 참조.

### 4.3 Review scope

매 cycle **전체 PR diff** (`origin/main...HEAD`). incremental scope 금지 — 기존 finding 사각지대 방지.

## 5. Fix cycle — REQUEST_CHANGES 처리

1+ reviewer 가 `REQUEST_CHANGES` 시 orchestrator (main session) 의 의무.

전체 순서 (위→아래 = 시간 순서. main session 이 모든 단계를 중계 — sub-agent 간 직접 hand-off 없음):

```mermaid
sequenceDiagram
    participant R as 4 Reviewer
    participant M as main session
    participant F as fix (executor)
    participant V as verifier
    R->>M: REQUEST_CHANGES
    loop reviewer별 순차 (도메인 모델 고정)
        M->>F: fix 위임
        F->>M: 완료 (working tree 편집, 개별 commit 안 함)
    end
    M->>M: fix 결과 commit
    M->>V: verify 위임
    V->>M: PASS / FAIL 보고
    Note over M: FAIL → 다시 fix (한도 초과 시 needs-human-review)<br/>PASS → push
    M->>R: safe-push → 4 reviewer 재검토
```

> 위 다이어그램은 §2 아키텍처 플로차트(전체 시스템 개요)와 **역할 분리**: 플로차트 = 무엇이 어디에, 본 시퀀스 = fix cycle 의 시간 순서.

### 5.1 Scope 판정

| 상황 | 처리 |
|---|---|
| **본 PR scope 안** + PR 크기 적정 | reviewer별 fix → 단일 verify → 본 PR 에서 commit (§5.2) |
| **본 PR scope 밖** 또는 **PR 비대** | GitHub Issue 등록 → 별도 follow-up PR |

### 5.2 Fix → Verify → Push (scope 안)

**fix** — REQUEST_CHANGES 낸 **reviewer 별 전용 fix sub-agent** (agent type `executor`) 를 **단일 worktree 에서 순차** 호출. 모델은 reviewer 도메인별 정적 고정 (런타임 escalate 판단 없음) — reviewer↔fix 모델 매핑은 [`fix-review/SKILL.md`](../../.claude/skills/fix-review/SKILL.md) 참조.

각 fix sub-agent 는 `fix-review` skill 의 `brief-template.md` 의무 항목 포함:

- review comment 직접 read (`gh pr view --json reviews`)
- 옛 fix evidence 읽기 (`.omc/state/fix-evidence/pr-<N>.json`) — 역추론 금지
- regression verify (evidence 의 fix_lines grep)
- finding 의 location + code field 읽기
- fix evidence 쓰기 (다음 cycle 사각지대 방지)

각 fix sub-agent 는 **working tree 만 편집하고 개별 commit 하지 않는다**. fix 수렴 후 main session 의 commit·push 규칙은 [`safe-push/SKILL.md`](../../.claude/skills/safe-push/SKILL.md) 참조.

**verify** — fix 수렴 후 **단일 verifier sub-agent** 가 변경 전체를 한 번에 검증하고 `PASS`/`FAIL` 로 main session 에 보고만 한다 (push 금지). verifier 구성·검증 의무·FAIL 기준은 [`verify-fix/SKILL.md`](../../.claude/skills/verify-fix/SKILL.md) 참조.

**push** — verify 결과별 main session 의 행동:

- `FAIL` → main session 이 재-fix 지시 (fix→verify 재실행). 재시도 한도는 §5.4 (정의: 스킬).
- `PASS` → main session 이 `safe-push` → 4 reviewer cycle 재진입

### 5.3 Follow-up Issue (scope 밖)

scope 밖 finding 또는 PR 이 너무 커지는 경우 — **무시 금지**, GitHub Issue 로 분리:

```bash
gh issue create --label follow-up,<severity> --title "..." --body "출처: PR #N, finding: ..."
```

본 PR description 의 `## Follow-up` 섹션에 issue link 명시. 본 PR 머지 후 별도 PR 로 처리.

> **핵심 원칙**: 어떤 WARN/MED finding 도 "무시" 로 끝나지 않는다. 본 PR 처리하거나 Issue 로 분리.

### 5.4 Retry 한도

두 루프(outer reviewer cycle · inner fix→verify) 모두 한도 초과 시 `needs-human-review` label + 사용자 개입 (orchestrator 자율 진행 중단). **각 한도 값·초과 조건**은 owning 스킬 참조:

- reviewer cycle (outer) → [`safe-push/SKILL.md`](../../.claude/skills/safe-push/SKILL.md)
- fix→verify (inner) → [`verify-fix/SKILL.md`](../../.claude/skills/verify-fix/SKILL.md)

## 9. 문서·스킬 소유권

모든 규칙은 **스킬이 정의**하고, 본 03 문서는 흐름·순서(WHAT/ORDER)만 기술하며 규칙은 **참조만** 한다. 같은 규칙을 두 곳에 실체 정의하면 한쪽만 수정될 때 silent drift 가 발생한다.

| 규칙 | 소유 스킬 | 03 에서 |
|---|---|---|
| verdict 계산 (role-specific blocking severity) | [`post-review/SKILL.md`](../../.claude/skills/post-review/SKILL.md) § verdict 자동 결정 (+ `post.sh`) | §4.2 참조 |
| fix 모델 매핑 (opus/sonnet) | [`fix-review/SKILL.md`](../../.claude/skills/fix-review/SKILL.md) | §5.2 참조 |
| verifier 구성·FAIL 기준·inner 재시도 한도 | [`verify-fix/SKILL.md`](../../.claude/skills/verify-fix/SKILL.md) | §5.2·§5.4 참조 |
| commit·push(재push 전략)·reviewer cycle(outer) 재시도 한도·reviewer spawn 의무 | [`safe-push/SKILL.md`](../../.claude/skills/safe-push/SKILL.md) | §5.2·§5.4 참조 |

> 새 규칙 추가 시: 스킬에 정의하고 03 은 링크만 둔다. 03 에서 규칙 값을 재서술하지 않는다.
